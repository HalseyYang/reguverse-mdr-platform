# ReguVerse 香港注册文件修订任务设计说明

日期：2026-07-22  
状态：用户已逐部分确认，等待书面设计审阅

## 1. 产品目标

在 ReguVerse MDR Platform 的项目工作台中增加“香港注册文件修订”任务。第一版处理一份用户上传的现有文件，按照香港医疗器械行政管理制度（MDACS）要求和 Bioray 模板完成合规修订、专业英文优化及结构重组，并输出带 Word 原生修订和 Bioray 批注的可编辑 DOCX。

该任务的最终产品是修订后的注册文件，而不是差距报告。网页审查摘要、规则检查和待确认项只用于支持文件修改与用户决策。

第一版不自动生成整套香港注册资料包，不自动替代法规专业判断，不直接向 MDIS 递交，也不输出 PDF。

## 2. ReguVerse 集成位置

香港注册文件修订是项目内部任务，不新增左侧独立香港注册板块。

“新建项目与设备画像”的“法规区域”增加完整选项：

`香港注册（MDACS）`

选择后：

- “器械类别”切换为香港类别；
- “分类规则”字段改为“香港分类依据”；
- 香港分类依据只允许选择一个；
- 系统根据分类依据自动推荐 Class II、Class III 或 Class IV；
- 用户可以修改建议类别；
- 类别与分类依据不一致时，系统显示明确警告，要求用户确认并记录原因；
- 页面实时摘要显示香港注册、分类依据和确认后的类别。

创建项目时，系统自动建立“香港注册文件修订”任务。除非项目同时选择 EU MDR 工作范围，否则不自动建立 Clinical Evaluation 任务。

## 3. 支持范围

### 3.1 输入格式

第一版支持：

- DOCX；
- 文字型 PDF；
- 扫描型 PDF；
- PNG；
- JPG。

第一版不支持 XLSX。

### 3.2 输出格式

所有可修订文件统一输出可编辑 DOCX，不生成 PDF。

- DOCX 输入保留原文对应关系并记录新增、删除和替换；
- PDF 和图片先提取或识别内容，再按照 Bioray 模板重建 DOCX；
- 所有草稿在文件名和正文中标明 `DRAFT + version`。

### 3.3 Bioray 模板

以下现有模板直接复用：

- MDS-01；
- MDS-02；
- 风险管理报告；
- 临床评价报告；
- 基本原则符合性检查表。

模板来源：

`E:\香港注册\hk-medical-device-submission\hk-medical-device-submission\templates`

现有模板未覆盖的文件类型，根据相应香港法规要求建立新的 Bioray 标准模板。新模板必须由用户确认后，才能写入模板目录并用于正式输出。不得把文件强行套入不匹配的模板。

### 3.4 第三方证据文件

证书、监管批准文件和第三方检测报告属于不可修改的证据文件：

- 原始正文不得修改或被系统覆盖；
- 系统检查有效期、签发机构、产品范围、制造商名称和香港注册适用性；
- 网页输出审查摘要和待办项；
- 需要翻译时，生成明确标注为英文翻译稿的 `DRAFT + version` DOCX；
- 翻译稿不得冒充原始证据。

## 4. 项目任务流程

香港注册文件修订任务分为六个连续阶段：

1. 上传文件；
2. 提取内容；
3. 确认文件类型与 Bioray 模板；
4. 香港要求审查与修订；
5. 待确认项；
6. DOCX 版本与下载。

任务首页显示：

- 当前上传文件；
- 建议文件类型；
- 对应香港注册项目；
- 使用模板；
- 当前处理阶段及章节进度；
- 当前草稿版本；
- 待确认项数量；
- 最新 DOCX 下载入口；
- 历史版本入口。

## 5. 文件处理流程

### 5.1 文件保存

每个任务一次只处理一个文件。系统保存原始文件并计算内容指纹，避免同一文件被重复上传。原始文件不被草稿或正式版覆盖。

上传大小限制必须在页面、前端、API、中间件和存储层保持一致。超限响应必须使用前端可识别的结构化错误，不能再次出现底层返回普通文本而前端强制解析 JSON 的问题。

### 5.2 内容提取

- DOCX：提取标题层级、段落、列表、表格、图片关系和原文顺序；
- 文字型 PDF：提取逐页文字和位置；
- 扫描型 PDF、PNG、JPG：执行文字识别，按页保存进度；
- 提取结果保留页码、段落或表格位置；
- 页面关闭后，重新进入可从已保存进度继续。

### 5.3 文件类型推荐

系统先显示：

- 建议文件类型；
- 对应香港注册项目；
- 推荐模板；
- 判断依据。

用户确认或修改文件类型后，才开始正式修订。人工智能不得直接决定目标模板并静默处理。

### 5.4 固定规则检查

固定规则先检查：

- 模板必需章节；
- 产品名称、型号、制造商和预期用途的一致性；
- 文件类型对应的香港必需字段；
- GN-02、TR-004、TR-005 及其他适用要求；
- 明确占位符；
- 设备画像中的分类依据和类别；
- 任何可能从其他项目、示例或模板继承的客户事实。

## 6. 人工智能分工

采用“结构化规则与人工智能分工处理”，不把整份大文件一次发送给一个模型。

### 6.1 模型分工

- `deepseek-v4-flash`：文件类型识别、结构提取和低风险格式检查；
- `deepseek-v4-pro`：法规修订、专业英文优化和修改说明。

### 6.2 分章节处理

- 文件按章节处理，网页显示完成章节数和当前章节；
- 每个章节只能读取该章节原文、当前项目已确认设备画像、已确认模板和适用香港要求；
- 人工智能可以改善英文、术语、结构、表格和交叉引用；
- 不得创造原文件或用户确认内容中不存在的技术事实；
- 缺失事实必须写为 `[TO BE PROVIDED: ...]`；
- 单个章节失败只重试该章节，不从头处理整份文件。

### 6.3 请求控制

每个请求设置明确超时、失败分类和重试边界。界面分别显示模型超时、限流、服务不可用、返回格式错误、来源验证失败和 DOCX 生成失败，不得无限停留在静态“处理中”状态。

## 7. DOCX 修订规则

### 7.1 草稿

第一份草稿示例：

`Risk_Management_Report_DRAFT_v0.1.docx`

草稿必须：

- 首页标明 `DRAFT_v0.1`；
- 按已确认的 Bioray 模板排版；
- 使用 Word 原生修订记录新增、删除和替换；
- 修订作者统一为 `Bioray`；
- 法规性修改使用 `Bioray` 批注，只注明法规文件名称，例如“依据 GN-02”；
- 待确认项和 `[TO BE PROVIDED: ...]` 使用 `Bioray` 批注；
- 保持所有可追溯的原始技术事实，不把示例或模型推断写成正式事实。

### 7.2 修改深度

系统同时完成：

- 香港注册合规修改；
- 专业英文优化；
- 医疗器械术语统一；
- 章节和表格重组；
- 交叉引用优化。

系统不得通过语言优化改变原技术事实、产品范围或法规结论。

### 7.3 正式版

所有开放项解决并通过质量检查后，生成不带 `DRAFT` 标识的正式最新版 DOCX。正式版不得包含未解决的 `[TO BE PROVIDED: ...]`。

## 8. 网页审查摘要和开放项

除 DOCX 外，网页显示：

- 文件类型和采用模板；
- 主要修改；
- 香港法规依据；
- 待确认项；
- 当前处理状态；
- 当前版本及历史版本。

需要用户拍板时，一次只显示一个开放项，固定采用：

1. 场景；
2. 冲突与问题；
3. 方案。

用户每确认一个实质事项，系统重新生成下一版草稿。推荐方案不等于用户已选择。

以下内容不得由人工智能自动确定并写入正式版：

- 预期用途、适应症和禁忌症的实质变化；
- 产品型号和注册范围；
- 制造商或生产地址；
- 香港器械类别与分类依据不一致；
- 技术性能参数；
- 风险可接受性和风险收益结论；
- 临床安全性、性能或有效性结论；
- 证书真实性、有效性和产品覆盖范围；
- 缺失的 LRP、批准、检测或签署信息。

## 9. 数据与版本

### 9.1 数据分层

每个任务分别保存：

- 原始上传文件；
- 提取文字、页码和结构；
- 用户确认的文件类型；
- 模板及模板版本；
- 固定规则检查结果；
- 各章节人工智能修订结果；
- 待确认项及用户回答；
- 每个 DOCX 草稿版本；
- 当前正式最新版；
- 操作和修改记录。

人工智能结果不得直接覆盖原始文件或已确认信息。

### 9.2 版本编号

- 第一份草稿：`DRAFT_v0.1`；
- 每次实质确认后：`DRAFT_v0.2`、`DRAFT_v0.3`；
- 没有实质修改时不增加版本；
- 全部确认后：正式版 `v1.0`；
- 正式版后再次修改：先生成 `DRAFT_v1.1`；
- 项目页面默认显示最新文件，历史版本可以查看和下载。

### 9.3 修改记录

每次实质修改记录：

- 修改前内容；
- 修改后内容；
- 修改人；
- 修改时间；
- 修改原因；
- 来源位置；
- 模板版本；
- 香港法规文件名称；
- 对应 DOCX 版本。

删除草稿不删除操作记录。用户删除整个项目时，按项目删除规则处理业务文件和数据。

### 9.4 本地目录

第一版沿用 ReguVerse 的本地数据目录，并增加：

```text
data/
  uploads/
  hong_kong_registration/
    <project_identifier>/
      source_files/
      extracted_content/
      draft_versions/
      final_versions/
      task_records/
```

客户文件不写入 Git，不提交到版本控制。

## 10. 异常恢复

- 文件提取、文字识别和章节修订分别保存进度；
- 关闭页面或重新启动服务后可继续；
- 单个章节失败只重试该章节；
- 失败不删除原文件、提取内容或已完成章节；
- 长时间“处理中”时，系统检查任务租约或最后心跳，显示“仍在运行”“已超时”或“可以重试”；
- 重试操作保持幂等，不得重复创建相同版本。

## 11. 权限

沿用 ReguVerse 项目权限并增加任务操作：

- 项目负责人：建立任务、上传文件、选择模板、管理版本、生成正式版和删除任务；
- 法规审核确认者：确认文件类型、处理开放项、批准正式版；
- 资料提供者：上传文件、补充信息、查看和下载草稿；
- 只读查看者：查看摘要、版本历史和已授权文件。

当前本地原型尚未启用登录时，本机用户默认具有项目负责人权限。数据结构仍保留角色和操作字段，以便后续启用团队账号。

## 12. 安全边界

- DeepSeek 密钥只存放于本地环境配置，不进入代码、网页、日志或 Git；
- 原始文件和 DOCX 版本只保存于本地项目数据目录；
- 普通日志不记录文件正文、识别文本、密钥或客户敏感文件名；
- 下载接口只允许访问当前项目登记的文件；
- 文件路径必须校验，禁止目录穿越；
- 同时校验文件扩展名、内容类型、文件签名和大小；
- 文档中的指令视为不可信内容，不能改变系统任务；
- 人工智能不得读取其他项目文件补齐当前文档；
- 每项新增事实必须来自当前原文或用户确认，否则转为占位符。

## 13. 本地长期运行

在 Windows 上增加：

- 用户登录后自动启动 ReguVerse 前端和 API；
- 启动前检查 5173 和 8787 端口，避免重复进程；
- 服务异常时可以恢复；
- 创建“打开 ReguVerse”桌面入口；
- 双击入口时检查服务、恢复服务并打开 `http://localhost:5173/`；
- 保存简洁日志，不记录密钥或客户文件内容。

香港注册任务直接写入 `E:\reguverse-mdr-platform`。原 `E:\香港注册\site` 暂不删除，仅保留为历史参考，不再继续扩展。

## 14. 测试要求

自动测试至少覆盖：

- 法规区域切换后的字段联动；
- 香港分类依据推荐类别及人工修改警告；
- 创建项目时自动生成香港注册文件修订任务；
- DOCX、文字 PDF、扫描 PDF、PNG、JPG；
- 损坏、加密和超大文件；
- 文件类型推荐和用户更改；
- 第三方证据只审查、不修改；
- 分章节处理、失败重试和断点继续；
- 未经来源支持的新增事实被拦截；
- Word 原生修订作者为 `Bioray`；
- Word 批注作者为 `Bioray`；
- `DRAFT + version` 命名和递增；
- 正式版不含 `DRAFT` 和未解决占位符；
- 历史版本可下载且最新版本正确展示；
- Windows 自动启动、端口检查和一键恢复；
- 现有 EU MDR 项目和测试不回归。

## 15. 首版验收场景

使用一份不含真实客户信息的合成英文风险管理报告：

1. 新建项目并选择`香港注册（MDACS）`；
2. 选择香港分类依据，系统推荐类别；
3. 创建“香港注册文件修订”任务；
4. 上传 DOCX；
5. 系统推荐“风险管理报告”和对应模板；
6. 用户确认；
7. 系统分章节审查并显示进度；
8. 生成 `DRAFT_v0.1` DOCX；
9. Word 中可见 `Bioray` 修订和批注；
10. 网页逐项处理待确认内容；
11. 生成后续草稿；
12. 解决全部开放项后生成正式 DOCX；
13. 重启电脑后仍能打开项目、历史版本和正式版。

只有上述流程通过，并且现有 EU MDR功能和测试没有受到破坏，才视为第一版完成。

## 16. 第一版不包含

- 从单个文件自动生成整套香港注册资料包；
- 自动向 MDIS 递交；
- PDF 输出；
- XLSX 输入；
- 修改第三方证据正文；
- 未经用户确认的新模板直接投入正式使用；
- 人工智能自动决定高风险法规或技术结论；
- 面向互联网的生产部署。
