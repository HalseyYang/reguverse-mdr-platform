## 2. 当前已确认的核心产品逻辑

医疗器械分类助手不应要求用户填写分类结论。用户只应输入产品事实，系统负责按目标市场法规和知识库资料生成分类建议。

### 用户输入：产品事实画像

应包括：

- 产品名称
- 预期用途
- 工作原理 / 作用机制
- 是否有源
- 是否侵入 / 外科侵入 / 植入
- 接触部位
- 接触时间
- 是否无菌
- 是否测量
- 是否软件 / AI
- 是否 IVD
- 是否输出能量
- 是否已有参考市场批准

### 系统输出：分类建议

应包括：

- 推荐分类
- 规则 / 路径
- 编码 / 编号
- 判断理由
- IMA 知识库依据
- 待核对动作
- 人工最终确认入口

关键原则：不要把结果字段放在输入区。例如 FDA Product Code、FDA Regulation Number、NMPA 管理类别、NMPA 分类编码、MDACS Class 都应由系统输出，而不是由用户预先填写。

## 3. 四个市场的分类输出要求

### 3.1 CE MDR 分类

系统应输出：

- MDR Class
- 适用 Rule，例如 Rule 5、Rule 6、Rule 7、Rule 8、Rule 9、Rule 10、Rule 11 等
- Rule 触发理由
- 候选规则
- 信息缺口
- 人工确认项

当前已有后端基础接口：

- `POST /api/classifiers/eu-mdr`

后续需要增强规则覆盖、候选规则排序和 UI 展示。

### 3.2 FDA 注册分类

用户不应填写：

- Product Code
- Regulation Number
- FDA Panel
- Submission Type

系统应通过产品事实 + IMA 知识库 / FDA 分类资料输出：

- 候选 Product Code
- Regulation Number
- Device Class
- Submission Type：510(k)、De Novo、PMA、Exempt
- Review Panel
- 是否需要 predicate device
- 待核对依据

### 3.3 NMPA 注册分类

用户不应填写：

- 管理类别
- 分类编码
- 分类目录结论

系统应通过产品事实 + IMA 知识库 / NMPA 分类目录输出：

- 推荐管理类别
- 分类编码
- 分类目录条目
- 注册 / 备案路径
- 是否需要分类界定
- 待确认项

### 3.4 香港 MDACS 分类

用户不应填写：

- MDACS Class A/B/C/D

系统应通过产品事实 + IMA 知识库输出：

- MDACS Class A/B/C/D
- 是否 GMD / IVD
- 分类依据
- 是否适用 listing
- 是否需要参考市场批准资料
- 香港注册文件要求
- 待确认项

## 4. IMA 知识库接入要求

用户明确要求：分类判断必须接入 IMA 知识库，否则判断不够标准。

当前系统已有后端基础：

- `POST /api/knowledge/search`
- 支持检索 IMA 知识库
- 支持多个法规知识库检索和结果汇总

当前应保留并使用的两个知识库：

- 医疗器械国际认证汇总知识库
- 医疗器械国内注册汇总知识库

### 目标链路

1. 用户输入产品事实
2. 系统根据当前市场构造检索 query
3. 调用 IMA 知识库检索
4. 基于命中资料生成候选分类
5. 输出依据和待确认项
6. 用户人工确认最终分类

### 重要边界

- IMA 是内部知识库依据，不等同于官方最终依据。
- 正式报告仍需记录官方来源、版本、日期和访问日期。
- 分类结果必须能追溯到命中的知识库资料。

## 6. 当前技术债务 / 风险

### 6.1 前端入口曾被临时精简

为了快速恢复预览，`src/main.jsx` 曾被替换为简化入口。

影响：

- AI 工具和设置页可用。
- 其它旧模块可能仍为占位或尚未完整接回。
- 后续需要逐块恢复：
  - 项目与任务
  - 文档生成
  - 法规知识库
  - 香港注册文件修订
  - 团队与权限
  - 系统架构

### 6.2 分类逻辑仍需模型结构化判断

当前逻辑主要是：

- 本地规则
- IMA 检索命中
- 前端汇总展示

更标准的下一步应是：

- IMA 检索结果 → DeepSeek / AI 模型 → 结构化分类 JSON
- 输出 confidence、source mapping、conflict items
- 支持人工确认 / 覆盖
- 生成审计记录

### 6.3 IMA 依据需要官方核对

IMA 可作为后备知识库和内部检索依据。正式结论必须补充：

- 官方来源
- 文件版本
- 发布 / 修订日期
- 访问日期
- 适用性说明

## 7. 建议 GitHub Issue 拆分

### Issue 1：重构医疗器械分类助手输入模型

目标：统一产品事实输入，不让用户填写分类结论字段。

验收：

- FDA 不要求用户填写 Product Code / Regulation Number。
- NMPA 不要求用户填写管理类别 / 分类编码。
- 香港不要求用户填写 MDACS Class。
- CE MDR 不要求用户手动选择 Rule。
- 用户只输入产品事实。

### Issue 2：接入 IMA 知识库作为分类依据

目标：分类前先检索 IMA。

验收：

- 点击“保存并开始分类”会调用 `/api/knowledge/search`。
- 结果显示 IMA 命中资料。
- 每条分类建议能显示来源。
- 无 IMA 命中时提示补充信息或检查配置。

### Issue 3：CE MDR 分类规则增强

目标：完善 MDR Annex VIII 分类规则。

验收：

- 支持 Rule 1–22 初筛。
- 输出 controlling rule。
- 输出 candidate rules。
- 输出 excluded rules。
- 输出 information gaps。
- 支持 Rule 9、Rule 11 等高频场景。

### Issue 4：FDA 分类引擎

目标：根据产品事实检索和推荐 FDA 分类。

验收：

- 输出候选 Product Code。
- 输出 Regulation Number。
- 输出 Device Class。
- 输出 Submission Type。
- 输出 Review Panel。
- 标记是否需要 predicate device。

### Issue 5：NMPA 分类引擎

目标：根据产品事实推荐中国分类目录条目。

验收：

- 输出管理类别。
- 输出分类编码。
- 输出分类目录条目。
- 输出注册 / 备案路径。
- 不确定时提示分类界定。

### Issue 6：香港 MDACS 分类引擎

目标：根据产品事实判断香港 MDACS 分类。

验收：

- 输出 Class A/B/C/D。
- 区分 GMD / IVD。
- 输出适用规则依据。
- 输出 listing 文件要求。
- 标记参考市场批准资料需求。

### Issue 7：恢复其它平台模块

目标：把临时简化入口期间被占位的模块逐步接回。

验收：

- 项目与任务可用。
- 法规知识库可用。
- 文档生成可用。
- 香港注册文件修订模块可用。
- 设置页保留模型配置。
